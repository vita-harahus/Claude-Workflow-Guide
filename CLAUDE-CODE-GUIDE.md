# Гайд роботи в Claude Code

**v2.0 · для команди.** Документ описує повний цикл роботи дизайнера в Claude Code: від налаштування
середовища до готових сторінок і передачі в розробку.

Документ поділено на три частини:

| Частина | Для кого | Зміст |
|---|---|---|
| **I. Для дизайнера** | 🧑‍🎨 дизайнер | Покроковий процес — що робити руками, від нуля до сторінок. |
| **II. Правила для Claude** | 🤖 Claude | Дизайн-система, що виноситься у `CLAUDE.md` і застосовується автоматично. |
| **III. Webflow-експорт** | 🧑‍🎨 + 🤖 | Правила й послідовність передачі проєкту в Webflow. |

# Частина I. Для дизайнера — покроковий процес

Дії, які дизайнер виконує самостійно. Кожен крок — окрема, завершена операція.

### Крок 1. Встановлення (одноразово)

Робота ведеться в **десктоп-застосунку Claude Code** або в терміналі. Веб-версія працює лише з
GitHub-репозиторієм у хмарі — не з локальною папкою на компʼютері.

```bash
npm install -g @anthropic-ai/claude-code
```

### Крок 2. Папка проєкту та матеріали клієнта

Створити на компʼютері папку проєкту й скласти в неї **всі вхідні матеріали**: бриф, референси, логотип,
шрифти, зображення, гайдлайни бренду і — за наявності — готову дизайн-систему.

```bash
mkdir ~/Desktop/nero-run        # → перенести у папку всі матеріали клієнта
```

### Крок 3. Синхронізація з Claude Code

Відкрити папку в застосунку (або в терміналі). Вона стає робочою директорією сесії, і Claude отримує
доступ до всіх матеріалів.

```bash
cd ~/Desktop/nero-run && claude
```

### Крок 4. Дизайн-система — один із двох шляхів

- **Завантажити наявну** — покласти файли дизайн-системи клієнта в папку; Claude працює за ними.
- **Створити з нуля** — Claude формує дизайн-систему за правилами Частини II. Першою збирається сторінка
  **Style Guide** (кольори, типографічна шкала, кнопки, лінки, картки) — це підтверджує, що система застосована.

### Крок 5. Локальний перегляд (localhost)

Одразу після появи перших сторінок піднімається локальний сервер — результат видно в браузері наживо, з
реальними зображеннями та анімаціями. Для цього дизайнер встановлює **два скіли**:

- **`local-preview`** — піднімає локальний HTTP-сервер і віддає адресу `http://localhost:<порт>` (перший запуск; працює на статичних `.html/.css/.js`).
- **`localhost-up`** — (пере)запускає сервер, коли localhost впав, зупинився або після рестарту сесії.

### Крок 6. Побудова сторінок посекційно

Робота ведеться з кодом, а не зі скріншотами. Точкові правки стосуються лише запитаного елемента.
Зображення підтягуються з реальних фотостоків через API (Pexels / Unsplash).

### Крок 7. Компоненти

Повторювані блоки (navbar, footer, картки, CTA) оголошуються **майстер-елементом** один раз — після цього
будь-яка зміна поширюється на всі інстанси на всіх сторінках. Детально — Частина II, розділ «Компоненти».

### Крок 8. Адаптивність

Перевіряються всі **7 контрольних точок** (1920 / 1440 / 1280 / 992 / 768 / 568 / 360); 0 помилок у консолі.

### Крок 9. Передача в розробку

За потреби проєкт передається у Webflow — послідовність і правила описані в **Частині III**.

> **Ключова умова.** У корені папки лежить файл `CLAUDE.md` із правилами Частини II. Claude зчитує його на
> старті сесії та дотримується правил автоматично.

# Частина II. Правила для Claude

Ці правила Claude застосовує самостійно (виносяться у `CLAUDE.md`). Дизайнер задає лише вихідні дані —
шрифти, кольори, компоненти — і не виконує ці пункти вручну.

## ⛔ Обовʼязкове: без AI-дефолтів, рівень Awwwards

Дизайн-система нижче є **технічним каркасом розмітки**, а не візуальним рецептом. Візуально кожен проєкт
має бути **унікальним і креативним, на рівні робіт Awwwards**.

**За замовчуванням не використовуються типові AI-елементи:**
- ❌ Eyebrow-сабтайтл із крапкою (`• LABEL`) над секціями — лише за прямим запитом дизайнера.
- ❌ Номери `01/02/03`, якщо контент не є справжньою послідовністю.
- ❌ Шаблонний hero «велика цифра + лейбл + градієнт».
- ❌ Стокові AI-палітри (кремовий + serif + теракота; чорний фон + кислотний акцент; «газетний» hairline-лейаут) — за відсутності відповідної вимоги брифу.
- ❌ Надмірна анімація, симетричні однотипні блоки, декор без функції.

**Натомість застосовуються принципи:** hero подає ключову тезу (найхарактерніше у сфері продукту);
типографіка формує характер сторінки; структурні елементи кодують зміст, а не декорують його; один виразний
**signature-елемент**, решта оформлення стримана; свідомий творчий ризик; базовий рівень якості (адаптивність,
видимий keyboard-focus, підтримка `prefers-reduced-motion`). Перед версткою складається короткий план (палітра,
шрифти, лейаут, signature) і звіряється з брифом: дефолтне рішення, придатне для будь-якої схожої сторінки,
переробляється.

## Брейкпоінти

| Брейкпоінт | Діапазон | Напрям |
|---|---|---|
| **Base** (default) | 992–1279 | базові стилі, каскад ↑↓ |
| Large / XL / XXL | 1280–1439 / 1440–1919 / 1920+ | ↑ up |
| Medium / Small / Tiny | 768–991 / 568–767 / 360–567 | ↓ down |

Стилі, задані на Base, каскадуються і вгору, і вниз. **Оверрайд застосовується лише там, де значення
відрізняється від Base.** Перевірці підлягають усі точки: **1920 / 1440 / 1280 / 992 / 768 / 568 / 360**.

## Структура секції (єдина для всіх сторінок)

```html
<section class="section section-name">
  <div class="base-container">
    <div class="section-head">
      <!-- eyebrow — лише за запитом дизайнера, не за замовчуванням -->
      <h2 class="heading-spacer-top">Section Title</h2>
      <p class="lead">Supporting description.</p>
    </div>
    <!-- section content -->
  </div>
</section>
```

- Секція = `<section>` + `section` + сабклас (`section trust`). Контент розміщується у `base-container`.
- **`base-container`:** `max-width` визначається дизайнером під проєкт (виноситься у змінну, єдину для всіх), `margin: auto`, `width: 100%`, `padding: 0 15px`.
- **Падінги секції задаються лише через `section` / `section-top` / `section-bot`** (шкала нижче), а не на власному класі секції.
- Темні секції: фон задається на класі секції; дочірні елементи отримують клас `on-dark`.

**Section Padding (перший клас секції):**

| Клас | Base | 1280+ | 768–991 | ≤767 |
|---|---|---|---|---|
| `section` | 100px 0 | 130px 0 | 80px 0 | 60px 0 |
| `section-top` | 100px 0 0 | 130px 0 0 | 80px 0 0 | 60px 0 0 |
| `section-bot` | 0 0 100px | 0 0 130px | 0 0 80px | 0 0 60px |

## Нейминг класів

- **lowercase-hyphen**, секційний префікс: `trust-head`, `approach-title`, `foot-grid`.
- **Максимум 2 класи**: структурний + модифікатор (`btn btn-primary`, `eyebrow on-dark`).
- Модифікатори без префікса (`on-dark`, `on-light`, `high`); стани — `is-*`; CMS-маркери — `field-*` (знімаються після біндингу).
- **Кожен елемент має власний клас; descendant-селектори не використовуються** (✅ `.approach-title` ❌ `.trust-head h2`). Інлайн-стилі не застосовуються.

Ключові утиліти: `on-dark`/`on-light`, `base-container`, `heading-spacer-top` (mt 22px),
`small-spacer-top` (mt 10px), `link-reset`, `relative` (z-index стек), `high` (акцент у заголовку).

## Кольори

**Значення відрізняються в кожному проєкті** (наведені нижче — приклад). Незмінною лишається **система**:
коректні назви ролей (Accent, Ink, Muted, Line Light…), уніфікація (один колір — одна змінна),
**перевірка контрасту за WCAG** (≥4.5:1 для основного тексту, ≥3:1 для великих заголовків/UI) на реальних поєднаннях.

Приклад ролей: `Accent #0162ff` · `Accent Light #eaf1ff` · `Accent Dark #0148bf` · `White #fff` ·
`Off White #FAFAF9` · `Ink #111` · `Muted rgba(17,17,17,.62)` · `Line Light rgba(17,17,17,.10)` · `Line Dark rgba(255,255,255,.16)`.

## Типографіка

**Шрифти визначає дизайнер під проєкт.** Обовʼязкові правила:
- **Максимум 2 шрифти**; **понад 3 шрифти не допускається** (за ознак четвертого роботу слід зупинити й уточнити).
- **Пріоритет — Google Fonts.** Для шрифтів поза Google Fonts спершу перевіряється ліцензія на web-embed або узгоджується з дизайнером; самостійне встановлення не допускається.
- Ролі: **Primary — заголовки H1–H6, лого, декор**; **Secondary — основний текст, UI, навігація, футер**.
- **Кнопки/лінки** — на розсуд дизайнера; за замовчуванням рекомендується **Secondary** (краща читабельність на малих розмірах і в інтерактиві).

Шкали (приклад): основний текст XXS 14 → XXL 24px; ваги 300–800; line-height 1em→1.55; letter-spacing −1.5→0px.

## Spacing / Radius

Spacing задається змінними з режимами по брейкпоінтах; текстовий margin біндиться на змінну **Spacing None (=0)**.
Radius: XS 4 (теги) · S 6 (інпути) · M 8 (картки) · L 10 (панелі) · XL 100px (пігулки — кнопки/бейджі).

## Кнопки, лінки та розмітка

**Набір кнопок і лінків визначає дизайнер під кожен UI**; незмінною лишається система:
- База `.btn` + один варіант, максимум 2 класи.
- **Кнопки оформлюються завжди як `<a>`** (не `<button>`); іконка й текст — окремими прямими дочірніми елементами; для тоглів застосовується `e.preventDefault()`.
- **Лінки не використовуються без класу**: призначається клас за призначенням (`more`, `explore`, `link-reset`, `card-link`, `foot-link`).
- **Усі відступи задаються через flex/grid-контейнери** (`grid-row-gap`/`grid-column-gap` на батьківському елементі), а **не** на самих текстах, лінках чи `<p>`. Текстовий елемент власних відступів не несе.

## ⛔ Компоненти — єдиний майстер-елемент

Якщо дизайнер визначає елемент як **Компонент** (navbar, footer, картка, кнопка, CTA…), це **єдиний
майстер-елемент**, ідентичний на всіх сторінках:
- Елемент оголошується майстер-елементом; реєструються **всі** місця його використання.
- **Зміна в одному місці автоматично поширюється на всі інстанси** на всіх сторінках. Ручне дублювання по сторінках не застосовується.
- Ідентичність повна (розмітка, класи, контент, стилі, стани); відрізнятися може лише свідомо винесене у проп/варіант.
- **Перед будь-якою правкою елемента перевіряється, чи не є він компонентом**; якщо так, зміни вносяться в майстер-елемент і поширюються на всі інстанси, а не точково на одній сторінці.

# Частина III. Webflow-експорт

Застосовується **лише під час передачі проєкту в Webflow**. Ці правила забезпечують коректний і чистий
експорт розмітки, класів, змінних і CMS.

## ⚠️ Без Custom properties (ключова умова коректного експорту)

Після передачі панель **Custom properties має лишатися порожньою**. Для цього:
- **CSS задається лише в longhand-форматі** (сторони border/padding/margin окремо, чотири радіуси окремо, `background-color`/`background-image` окремо).
- **Використовується `grid-row-gap`/`grid-column-gap`; властивість `gap` не застосовується.**
- **Градієнтні/сіткові фони** (`linear-gradient` + `background-size`) виносяться у **Site Head Code**, а не задаються на класі.
- Кастомні властивості на класах не застосовуються.

## Анімації — через атрибути (обробляються IX2/GSAP)

- `data-load="fade-in"` — елементи над згином (hero, navbar), поява на завантаженні.
- `data-scroll="fade-in"` — кожна секція, поява на скролі.
- `data-scroll="fade-in-item"` — контент-блоки всередині (картки, колонки, заголовковий wrap; сітки — покартково/стаггер).
  Не застосовується до: секцій, контейнерів, декору, «голих» заголовків, глибоких вузлів.

## Передача через розширення-білдер

Брейкпоінти (7 шт.) і Variables (кольори, шрифти, spacing, radius) налаштовуються у Webflow до початку
верстки. Компонентам відповідають **Symbols/Components** (навбар/футер лишаються як Symbol).

Послідовність CMS-флоу: **Annotate** (`field-[slug]`) → **Build + Apply** → назва колекції + slug →
**Scan Fields** → типи полів (Plain/Rich/Image) → **Create Schema** → **Load Items** → **Push Items** →
**Bind Fields** → **Remove Markers**.

**Не відновлюється через API** (виконується в Designer вручну): привʼязка CMS List до Source; мегаменю/дропдауни;
градієнтні фони (виносяться у Site Head Code); реальні Input/Form. Під час передачі змінні біндяться: кольори за
значенням, розміри ±1px із розділенням груп (font-size → Typography, spacing → Spacing).
