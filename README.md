<a id="0"></a>
## 0. ⛔ ОБОВʼЯЗКОВЕ: без AI-дефолтів, рівень Awwwards

Дизайн-система, наведена нижче, є **технічним каркасом розмітки**, а не візуальним рецептом. Візуально кожен
проєкт має бути **унікальним і креативним, на рівні робіт Awwwards**. Для кожного нового UI активується скіл
**`frontend-design`**.

**За замовчуванням не використовуються типові AI-елементи:**
- ❌ Eyebrow-сабтайтл із крапкою (`• LABEL`) над секціями — застосовується **лише за прямим запитом дизайнера**.
- ❌ Номери `01/02/03`, якщо контент не є справжньою послідовністю.
- ❌ Шаблонний hero «велика цифра + лейбл + градієнт».
- ❌ Стокові AI-палітри (кремовий + serif + теракота; чорний фон + кислотний акцент; «газетний» hairline-лейаут) — за відсутності відповідної вимоги брифу.
- ❌ Надмірна анімація, симетричні однотипні блоки, декор без функції.

**Натомість застосовуються такі принципи:** hero подає ключову тезу (найхарактерніше у сфері продукту);
типографіка формує характер сторінки; структурні елементи (eyebrow, номери, лейбли) кодують зміст, а не
декорують його; передбачається один виразний **signature-елемент**, решта оформлення стримана; свідомий
творчий ризик; дотримання базового рівня якості (адаптивність, видимий keyboard-focus, підтримка
`prefers-reduced-motion`). Перед версткою складається короткий план (палітра, шрифти, лейаут, signature) і
звіряється з брифом: якщо певна частина є дефолтним рішенням, придатним для будь-якої схожої сторінки, її
необхідно переробити.

## 1. Налаштування середовища

**Середовище запуску.** Для роботи з локальною папкою використовується Claude Code у терміналі або
десктоп-застосунку (веб-версія працює лише з GitHub-репозиторієм у хмарі):
```bash
npm install -g @anthropic-ai/claude-code
mkdir ~/Desktop/new-project && cd ~/Desktop/new-project && claude
```

**Конектори (MCP)** — підключаються командою `/mcp` з подальшою авторизацією:
- **Webflow** (обовʼязково) — передача елементів, класів і CMS безпосередньо в Designer.
- **Vercel** (рекомендовано) — превʼю-посилання. **GitHub** (опційно) — версіонування. **Higgsfield** — генерація зображень.

**Скіли** встановлюються **глобально** (діють у всіх проєктах, встановлюються один раз на пристрої, поза git проєкту):
```bash
npm i -g ui-ux-pro-max-cli && uipro init --ai claude --global   # → ~/.claude/skills/
# frontend-design → скопіювати в ~/.claude/skills/frontend-design/
```
Рекомендовані скіли: `ui-ux-pro-max` (стилі, палітри, шрифти, чарти), `frontend-design` (унікальний дизайн, розділ 0),
`webflow-converter`, `webflow-cms-collections`, `component-sync` (компоненти, 2.10), `design-memory`
(контекст між сесіями), `design-prompt-generator`, `image-sourcing`.

> **Ключова вимога.** У корені проєкту розміщується файл `CLAUDE.md` із правилами розділу 2 — Claude зчитує
> його на старті сесії та дотримується правил автоматично.

## 2. Дизайн-система (розміщується у `CLAUDE.md`)

### 2.1 Брейкпоінти (Webflow → Project Settings → Breakpoints, до початку верстки)

| Брейкпоінт | Діапазон | Напрям |
|---|---|---|
| **Base** (default) | 992–1279 | базові стилі, каскад ↑↓ |
| Large / XL / XXL | 1280–1439 / 1440–1919 / 1920+ | ↑ up |
| Medium / Small / Tiny | 768–991 / 568–767 / 360–567 | ↓ down |

Стилі, задані на Base, каскадуються і вгору, і вниз. **Оверрайд застосовується лише там, де значення
відрізняється від Base.** Перевірці підлягають усі точки: **1920 / 1440 / 1280 / 992 / 768 / 568 / 360**.

### 2.2 Структура секції (єдина для всіх сторінок)

```html
<section class="section section-name">
  <div class="base-container">
    <div class="section-head">
      <!-- eyebrow — лише за запитом дизайнера (розділ 0), не за замовчуванням -->
      <h2 class="heading-spacer-top">Section Title</h2>
      <p class="lead">Supporting description.</p>
    </div>
    <!-- section content -->
  </div>
</section>
```

- Секція = `<section>` + `section` + сабклас (`section trust`). Контент розміщується у `base-container`.
- **`base-container`:** `max-width` визначається дизайнером під проєкт (виноситься у змінну, єдину для всіх), `margin: auto`, `width: 100%`, `padding: 0 15px`.
- **Падінги секції задаються лише через `section` / `section-top` / `section-bot`** (шкала нижче), а не на власному класі секції.
- Темні секції: фон задається на класі секції; дочірні елементи отримують клас `on-dark`. Статичні секції (CTA, Footer) до CMS-колекції не належать.

**Section Padding (перший клас секції):**

| Клас | Base | 1280+ | 768–991 | ≤767 |
|---|---|---|---|---|
| `section` | 100px 0 | 130px 0 | 80px 0 | 60px 0 |
| `section-top` | 100px 0 0 | 130px 0 0 | 80px 0 0 | 60px 0 0 |
| `section-bot` | 0 0 100px | 0 0 130px | 0 0 80px | 0 0 60px |

### 2.3 Нейминг класів

- **lowercase-hyphen**, секційний префікс: `trust-head`, `approach-title`, `foot-grid`.
- **Максимум 2 класи**: структурний + модифікатор (`btn btn-primary`, `eyebrow on-dark`).
- Модифікатори без префікса (`on-dark`, `on-light`, `high`); стани — `is-*`; CMS-маркери — `field-*` (знімаються після біндингу).
- **Кожен елемент має власний клас; descendant-селектори не використовуються** (✅ `.approach-title` ❌ `.trust-head h2`). Інлайн-стилі не застосовуються.

Ключові утиліти: `on-dark`/`on-light` (світлі/темні тони), `base-container`, `heading-spacer-top` (mt 22px),
`small-spacer-top` (mt 10px), `link-reset`, `relative` (z-index стек), `high` (акцент у заголовку).

### 2.4 Кольори — Color Variables

**Значення відрізняються в кожному проєкті** (наведені нижче — приклад). Незмінною лишається **система**:
коректні назви ролей (Accent, Ink, Muted, Line Light…), уніфікація (один колір — одна змінна),
**перевірка контрасту за WCAG** (≥4.5:1 для основного тексту, ≥3:1 для великих заголовків/UI) на реальних поєднаннях.

Приклад ролей: `Accent #0162ff` · `Accent Light #eaf1ff` · `Accent Dark #0148bf` · `White #fff` ·
`Off White #FAFAF9` · `Ink #111` · `Muted rgba(17,17,17,.62)` · `Line Light rgba(17,17,17,.10)` · `Line Dark rgba(255,255,255,.16)`.

### 2.5 Типографіка — Font Variables

**Шрифти визначає дизайнер під проєкт.** Обовʼязкові правила:
- **Максимум 2 шрифти**; **використання понад 3 шрифти не допускається** (за ознак четвертого роботу слід зупинити й уточнити).
- **Пріоритет — Google Fonts.** Для шрифтів поза Google Fonts спершу перевіряється ліцензія на web-embed або узгоджується з дизайнером; самостійне встановлення не допускається.
- Ролі: **Primary — заголовки H1–H6, лого, декор**; **Secondary — основний текст, UI, навігація, футер**.
- **Кнопки/лінки** — на розсуд дизайнера; за замовчуванням рекомендується **Secondary** (краща читабельність на малих розмірах і в інтерактиві).

Шкали (приклад): основний текст XXS 14 → XXL 24px; ваги 300–800; line-height 1em→1.55; letter-spacing −1.5→0px.
Заголовкові розміри задаються як Variables з режимами по брейкпоінтах (оверрайд лише там, де значення відрізняється від Base).

### 2.6 Spacing / Radius
Spacing задається як Variables з режимами по брейкпоінтах; текстовий margin біндиться на змінну **Spacing None (=0)**.
Radius: XS 4 (теги) · S 6 (інпути) · M 8 (картки) · L 10 (панелі) · XL 100px (пігулки — кнопки/бейджі).

### 2.7 Кнопки, лінки та розмітка

**Набір кнопок і лінків визначає дизайнер під кожен UI**; незмінною лишається система:
- База `.btn` + один варіант, максимум 2 класи; hover/стани задаються у Webflow States panel.
- **Кнопки оформлюються завжди як `<a>`** (не `<button>`); іконка й текст — окремими прямими дочірніми елементами; для тоглів застосовується `e.preventDefault()`.
- **Лінки не використовуються без класу**: призначається клас за призначенням (`more`, `explore`, `link-reset`, `card-link`, `foot-link`).
- **Інпути оформлюються як `<div>`-плейсхолдер**; **форми — як `<div class="signup-form">`** (не `<form>`); реальні Input/Form збираються в Designer.
- Кожен декоративний `<svg>` має атрибут **`data-icon="…"`**. Тег `<br>` не використовується.
- **Усі відступи задаються через flex/grid-контейнери** (`grid-row-gap`/`grid-column-gap` на батьківському елементі), а **не** на самих текстах, лінках чи `<p>`. Текстовий елемент власних відступів не несе — відступ задається на обгортці (`*-wrap`/`section-head`).

### 2.8 ⚠️ Без Custom properties (ключова умова коректного експорту)
Після передачі панель **Custom properties має лишатися порожньою**. Для цього:
- **CSS задається лише в longhand-форматі** (сторони border/padding/margin окремо, чотири радіуси окремо, `background-color`/`background-image` окремо).
- **Використовується `grid-row-gap`/`grid-column-gap`; властивість `gap` не застосовується.**
- **Градієнтні/сіткові фони** (`linear-gradient` + `background-size`) виносяться у **Site Head Code**, а не задаються на класі.
- Кастомні властивості на класах не застосовуються.

### 2.9 Анімації — через атрибути (обробляються IX2/GSAP)
- `data-load="fade-in"` — елементи над згином (hero, navbar), поява на завантаженні.
- `data-scroll="fade-in"` — кожна секція, поява на скролі.
- `data-scroll="fade-in-item"` — контент-блоки всередині (картки, колонки, заголовковий wrap; сітки — покартково/стаггер).
  Не застосовується до: секцій, контейнерів, декору, «голих» заголовків, глибоких вузлів.

### 2.10 ⛔ ОБОВʼЯЗКОВЕ: Компоненти — єдиний майстер-елемент
Якщо дизайнер визначає елемент як **Компонент** (navbar, footer, картка, кнопка, CTA…), це **єдиний майстер-елемент**, ідентичний на всіх сторінках:
- Елемент оголошується майстром через **`component-sync`**; реєструються **всі** місця його використання.
- **Зміна в одному місці автоматично поширюється на всі інстанси** на всіх сторінках. Ручне дублювання по сторінках не застосовується.
- Ідентичність повна (розмітка, класи, контент, стилі, стани); відрізнятися може лише свідомо винесене у проп/варіант.
- **Перед будь-якою правкою елемента перевіряється, чи не є він компонентом**; якщо так, зміни вносяться через `component-sync`.
- У Webflow відповідає **Symbols/Components** (навбар/футер лишаються як Symbol).

## 3. Процес побудови проєкту

0. **Фундамент:** папка проєкту + `CLAUDE.md` (розділ 2); підключення конекторів і скілів; у Webflow — 7 брейкпоінтів та Variables (кольори, шрифти, spacing, radius).
1. **Сторінка Style Guide (першою)** — кольори, типографічна шкала, кнопки, лінки, картки оформлюються живими елементами з коректними класами (перевірка, що дизайн-система застосована).
2. **Компоненти** (правило 2.10) — повторювані блоки (navbar, footer, картки, CTA) збираються один раз → `component-sync` → реєстрація всіх інстансів. Навбар і дропдауни збираються як Symbol у Designer вручну.
3. **Побудова сторінок посекційно** — робота ведеться з кодом, а не зі скріншотами; застосовується шаблон 2.2 та нейминг 2.3. Точкові правки стосуються лише запитаного. Зображення — `image-sourcing`.
4. **Адаптивність** — усі 7 точок; оверрайд лише там, де значення відрізняється від Base; 0 помилок у console.
5. **CMS** — динамічні елементи позначаються `field-[slug]`; статичні (CTA, «Explore more», Footer, Nav) не біндяться. Колекції — `webflow-cms-collections`.
6. **Анімації** — атрибути з розділу 2.9.
7. **Передача / превʼю** — (опційно) превʼю-посилання через Vercel; Handoff-консоль (Copy CSS/HTML посекційно) регенерується завжди з робочого файлу.

**Приклад індексу секцій:** Hero `hero on-dark` · Trust `section trust` · Approach `section approach on-dark` ·
Platform `section-top platform` · Products/Industries `section-top …` · Testimonials `section testimonials` ·
Final CTA + Footer `cta-footer on-dark` (div-wrapper, не `<section>`).

<a id="4"></a>
## 4. Передача в розробку — Webflow (розширення-білдер)

Послідовність CMS-флоу (Step 4): **Annotate** (`field-[slug]`) → **Build + Apply** → назва колекції + slug →
**Scan Fields** → типи полів (Plain/Rich/Image) → **Create Schema** → **Load Items** → **Push Items** →
**Bind Fields** → **Remove Markers**.

**Не відновлюється через API** (виконується в Designer вручну): привʼязка CMS List до Source; мегаменю/дропдауни;
градієнтні фони (виносяться у Site Head Code); реальні Input/Form. Під час передачі змінні біндяться: кольори за
значенням, розміри ±1px із розділенням груп (font-size → Typography, spacing → Spacing).
